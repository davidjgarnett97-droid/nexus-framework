Nexus — Structural Architecture

How Operators, Domains, Projects, and Workflows fit together · 2026-06-23

Level 1 — Outermost
Operator
A person or organization with their own isolated deployment. Has their own credentials, users, and domain stack. Physical isolation — no cross-operator access.
Level 2
Domain
A functional area of the operator's life or business. Each domain has a directive, a set of skills, and an assigned agent. Domains can be added or removed without touching the core.
Level 3
Project
A specific initiative within a domain. Has its own context file, goal, status, and execution scripts. Projects are the unit of work the agents track and report on.
Level 4 — Innermost
Workflow
A deterministic Python script that does one thing reliably. Connects to real tools (Gmail, Notion, Redfin). The agent calls these — it never does the work itself.

How to Expand — Adding at Every Level
Add an Operator
Onboard a new person or org
1
Clone coreDeploy the shared source repo to their server
2
Configure domainsSelect which domains they need from the library
3
Auth connectorsOAuth Gmail, Notion, Calendar — one-time per tool
4
Create user profilesIssue capability tokens — Operational, Builder, or Admin
5
Run smoke testConfirm one workflow fires end-to-end, audit log captures it
Add a Domain
New functional area for any operator
1
Write the directivePlain-English SOP: what the domain does, what it outputs, edge cases
2
Create domain folderdomains/<name>/ + execution/<name>/ + directives/<name>/
3
Register skillsAdd entries to skill_registry.yaml with capability requirements
4
Assign an agentMap a sub-agent to the domain in the routing guide
5
Issue tokensGrant domain capability to the users who need it
Add a Project
New initiative within an existing domain
1
Create project folderdomains/<domain>/projects/<name>/
2
Write project_context.mdGoal, status, what's blocking, what comes next
3
Add execution scriptsDrop .py scripts into execution/<domain>/ for this project's workflows
4
Register any new skillsIf the project introduces new capabilities, add to registry
5
Domain agent picks it upNexus reads the project context — no additional routing needed
Add an Agent
New specialist to the system
1
Draft system promptUse the 11-section Universal Agent Template in MetaAgent.md
2
Define skill setList which skills/capabilities the agent can invoke
3
Add to skill_registry.yamlRegister the agent and its capability token
4
Update routing guideAdd to Nexus's CLAUDE.md sub-agent routing table
🔐 All new agents require HITL review before live deployment. Meta Agent proposes — Admin approves.
